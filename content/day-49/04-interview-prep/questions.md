# Day 49 — GenAI Production Patterns: Interview Q&A

---

## Q1: How would you evaluate a RAG system in production without hand-labeled ground truth?

**A:** Three automated metrics that don't require ground truth: (1) **Faithfulness** — decompose the answer into individual factual claims (using an LLM) and check each claim against the retrieved context. Score = fraction of claims supported. Doesn't require knowing the "correct" answer; (2) **Context Precision** — for each retrieved chunk, use an LLM to judge whether it's relevant to the user's question. High precision = retrieved chunks are mostly relevant, low precision = too much noise in context; (3) **Answer Relevancy** — embed the answer and generate 3-5 questions that the answer would answer, then compute semantic similarity between those generated questions and the original question. High similarity = the answer stays on topic. I run these on 10% of production traffic (sampled asynchronously to avoid latency impact) and track weekly averages. Alerts fire if faithfulness drops below 3.5/5. For periodic validation I maintain a manually labeled "gold set" of 100-200 questions with correct answers, run full RAGAS including Context Recall, and compare week-over-week.

## Q2: What is LLM-as-judge evaluation? What are its limitations and how do you mitigate them?

**A:** LLM-as-judge uses a strong model (GPT-4o) to evaluate another model's outputs against a rubric. It's scalable (no human annotation required), fast, and surprisingly accurate for well-defined criteria like faithfulness and relevance. Known biases: (1) **Verbosity bias** — judges tend to prefer longer, more detailed answers even when concise answers are correct; mitigate by adding "brevity is a virtue" to the rubric; (2) **Positional bias** — when comparing two answers, the first-presented one gets higher scores; mitigate by running each comparison twice with order swapped and averaging; (3) **Self-enhancement bias** — GPT-4 rates GPT-4 outputs higher; use diverse judge models or Anthropic Claude as judge for GPT-4 outputs; (4) **Calibration drift** — the judge's interpretation of "5/5" can shift; mitigate by anchoring the rubric with concrete examples of 1, 3, and 5 scores. Despite limitations, LLM-as-judge is 70-90% correlated with human judgments for most NLP tasks — sufficient for CI regression testing and monitoring trends.

## Q3: Walk me through your approach to prompt regression testing in a CI/CD pipeline.

**A:** Treat prompts as first-class code artifacts. The pipeline: (1) **Test suite** — maintain 50-100 labeled input/expected output pairs per prompt. Cover: typical cases, edge cases, failure modes, adversarial inputs; (2) **Evaluation function** — for each test case, run the new prompt and compare output to expected using task-appropriate metrics (exact match for classification, LLM-judge score for generation, schema validation for extraction); (3) **CI integration** — on every PR that modifies a prompt file, trigger the test suite. Set a minimum pass rate threshold (e.g., 95% for classification, 80% for generation — generation is harder to evaluate); (4) **Baseline comparison** — compare new prompt's score to the production prompt's score on the same suite. Require improvement or equal performance; (5) **Canary deployment** — even if tests pass, route 5% of production traffic to the new prompt for 24 hours before full rollout. Track user feedback (thumbs up/down) during canary. The test suite itself needs maintenance: add new test cases when you encounter new failure modes in production, and retire outdated test cases as the product evolves.

## Q4: How do you reduce AI feature costs by 50% without degrading quality?

**A:** A layered approach targeting different cost drivers: (1) **Model routing** (biggest impact, 30-60% cost reduction) — route simple queries to GPT-4o-mini (28x cheaper than GPT-4o). Classify by message length, keyword signals, complexity patterns. Verify quality with LLM-as-judge on a sample; (2) **Semantic caching** (10-30% additional reduction) — cache responses for semantically similar queries. Typical cache hit rate: 20-40% for customer support bots, lower for open-ended chat. Threshold tuning is critical; (3) **Batch API** (50% on eligible requests) — for non-real-time tasks (nightly summaries, data enrichment, offline analysis), use OpenAI Batch API at 50% discount; (4) **Prompt compression** (10-20% input token reduction) — summarize long contexts before injection, truncate conversation history aggressively, use sentence-level retrieval (precise chunks) instead of paragraph-level; (5) **Output token control** — set appropriate max_tokens for each endpoint; many production prompts have unnecessarily high max_tokens. Monitor actual output lengths and set max_tokens to p99 + 20% buffer. Combined effect: routing + caching + Batch API can achieve 50-70% cost reduction with minimal quality impact.

## Q5: What monitoring would you set up for a production AI feature launching next week?

**A:** Four categories of monitoring: (1) **Technical health** — API error rate (alert >1%), latency p50/p95/p99 (alert p95 > 8s), OpenAI 429 rate (alert >5%), and model fallback trigger rate; (2) **Cost** — daily token spend by model, cost per user, cost per feature, week-over-week growth rate (alert if >30% spike); (3) **Quality** — automated faithfulness score (LLM-as-judge on 10% sample), user satisfaction (thumbs up/down rate), session abandonment rate after AI response, and repeat queries (user asking same question repeatedly = answer wasn't helpful); (4) **Safety** — moderation API flag rate (input and output), prompt injection detection rate, unusual user behavior patterns (very long inputs, rapid-fire queries). Dashboard: DataDog or Grafana with: latency heatmap, cost burn chart, quality score trend, error rate time series. Alerts to PagerDuty for: error rate >1% sustained 5 minutes, cost 120% of daily budget, quality score drop >15% from baseline. First week post-launch: manual review of a sample of conversations daily to catch issues automated metrics miss.

## Q6: How would you implement A/B testing for a prompt change?

**A:** A/B testing a prompt requires three components: (1) **Traffic splitting** — use a deterministic hash of user_id modulo 100 to assign users to groups (not random per-request, which would cause inconsistent experiences). For 5% canary: hash < 5 → new prompt, else old prompt. Store the assignment in Redis for consistency; (2) **Metric collection** — log prompt_version with every request. Track per-version: response quality score (LLM-judge), user feedback rate, task completion rate (did they ask a follow-up question within 30 seconds? — suggests the first answer was insufficient), session length, and cost per interaction; (3) **Statistical significance** — run for sufficient duration and sample size before declaring a winner. For binary metrics (thumbs up/down): chi-squared test with p < 0.05 and at least 200 samples per variant. For continuous metrics (quality scores): t-test or Mann-Whitney U. Minimum detectable effect: define upfront what improvement matters (e.g., >5% improvement in quality score). Practical addition: monitor the test for unexpected negative effects in the first 24 hours (high error rate, cost explosion) with automatic rollback if triggered.

## Q7: What's the difference between observability for traditional APIs and AI-powered APIs?

**A:** Traditional API observability: request rate, latency, error rate, and status codes. These are deterministic — the same input always produces the same output, so you only need to monitor whether it's responding at all. AI API observability requires additional dimensions: (1) **Output quality** — the API might respond 200 OK but give a wrong, hallucinated, or unhelpful answer. Traditional monitoring doesn't catch this. Requires LLM-as-judge sampling; (2) **Semantic drift** — model behavior can change without any code change (OpenAI updates model weights, prompt interpretation shifts). Monitor output quality over time, not just uptime; (3) **Cost as a first-class metric** — API calls have wildly varying costs depending on prompt length and response length. Monitor cost-per-call distribution, not just total cost; (4) **Prompt versioning** — need to correlate quality metrics with the specific prompt version that generated each response; (5) **User behavior signals** — follow-up questions, regeneration requests, and session abandonment are quality signals invisible in traditional logs. Tools like Langfuse and LangSmith are purpose-built for AI observability and capture these dimensions that traditional APM tools (DataDog, NewRelic) miss.

## Q8: How do you handle model drift in production? How do you detect it and respond?

**A:** Model drift in AI is different from traditional ML drift. Two types: (1) **OpenAI model drift** — OpenAI updates model weights behind the same API endpoint (e.g., "gpt-4o" points to different versions over time). Your prompts might have been tuned for the old version; (2) **Data distribution drift** — the questions users ask evolve over time, drifting away from what your prompts were optimized for. Detection: run your regression test suite (50-100 labeled examples) weekly against production. If pass rate drops by >10% without a prompt change, investigate model drift. Monitor weekly rolling averages of LLM-judge quality scores — compare current week vs 4 weeks ago. Sudden drops indicate model updates; gradual drops indicate data drift. Response to model drift: (1) Re-run your prompt optimization process on the new model version; (2) Pin to a specific model version if available (`gpt-4o-2024-08-06` instead of `gpt-4o`); (3) Update few-shot examples if the model's interpretation of tasks has shifted. Response to data drift: analyze new query types appearing in production, add test cases for them, update prompts to handle new patterns.
